## 1. 용어 변경 (법적 이슈 방지)

### 1.1 용어 매핑 테이블

| 기존 용어 | 변경 후 | 영문 | DB 컬럼명 | 설명 |
|----------|--------|------|----------|------|
| **계모임** | **챌린지** | Challenge | `challenge` (기존: `gye`) | 그룹 채널 개념 |
| **계주** | **리더** | Leader | `leader_id` (기존: `creator_id`) | 챌린지 생성자 |
| **계원** | **팔로워** | Follower | `follower` (기존: `member`) | 챌린지 참여자 |
| **결제** | **리워드** | Reward | `reward` (기존: `payment`) | 지출/수익 개념 |
| **모임** | **정기 모임** | Regular Meeting | `meeting` | 실제 오프라인 이벤트 |
| **회비** | **참가비** | Participation Fee | `participation_fee` (기존: `monthly_fee`) | 월 납입금 |
| **보증금** | **보증금** | Deposit | `deposit` | 변경 없음 |

### 1.2 DB 테이블 리네이밍 전략

**옵션 A: 별칭(Alias) 사용 (권장)**
```sql
-- View 생성으로 기존 코드 영향 최소화
CREATE VIEW challenges AS SELECT * FROM gye;
CREATE VIEW challenge_followers AS SELECT * FROM gye_members;
CREATE VIEW rewards AS SELECT * FROM ledger_entries WHERE type IN ('EXPENSE', 'INCOME');

-- 장점: 기존 코드 수정 불필요, 빠른 적용
-- 단점: 성능 미세 오버헤드
```

**옵션 B: 실제 테이블 리네이밍 (Phase 2)**
```sql
-- 추후 리팩토링 시
ALTER TABLE gye RENAME TO challenges;
ALTER TABLE gye_members RENAME TO challenge_followers;

-- 장점: 완전한 마이그레이션
-- 단점: 모든 코드 수정 필요 (5-7일 소요)
```

**권장**: **옵션 A** (View 별칭) → Demo Day 후 옵션 B로 마이그레이션

### 2.2 핵심 비즈니스 규칙

#### 규칙 1: 정기 모임 생성 권한
- **리더만** 정기 모임 생성 투표를 발의할 수 있음
- 팔로워는 투표에만 참여 가능

#### 규칙 2: 리워드 투표 참가자
**옵션 A: 전체 팔로워 투표 (권장)**
- 챌린지 금고는 전체 팔로워의 공동 자산
- 지출 결정은 전체가 참여해야 공평

**옵션 B: 정기 모임 참석자만 투표**
- 실제 비용을 발생시킨 사람만 투표
- 불참자는 투표 불가

**권장**: **옵션 A** (전체 팔로워 투표)

#### 규칙 3: 정기 모임 참석 관리
- 정기 모임 확정 후 팔로워들이 자유롭게 참석/불참 선택
- 참석 인원 제한 가능 (선착순)
- 불참해도 패널티 없음 (단, 리워드 투표는 참여 가능)

### 3.2 구체적 AI 활용 사례

#### 사례 1: ERD 설계 (Claude Code)
**기존 방식 (3일)**:
1. 수작업으로 테이블 구조 설계 (1일)
2. DDL 작성 및 제약조건 추가 (1일)
3. 인덱스 최적화 (1일)

**AI 방식 (0.5일)**:
```
Prompt: "챌린지 시스템에 정기 모임(MEETINGS) 테이블을 추가해줘.
요구사항:
- 챌린지와 1:N 관계
- 날짜, 장소, 비용, 참석자 관리
- 투표(VOTES)와 연결
- Oracle DB 기준으로 DDL 생성"

→ Claude가 5분 안에 완전한 DDL + 인덱스 + 제약조건 생성
```

#### 사례 2: Spring Boot Service (Cursor IDE)
**기존 방식 (10일)**:
- MeetingService 작성 (3일)
- 트랜잭션 로직 (2일)
- 예외 처리 (1일)
- Strategy 패턴 구현 (4일)

**AI 방식 (3일)**:
```

"MeetingCreateVoteStrategy를 구현해줘.
ExpenseVoteStrategy와 같은 패턴으로:
1. Vote 승인 시 Meeting 생성
2. 투표 발의자를 자동 참석자로 등록
3. 트랜잭션 안전성 보장"

→ Cursor가 80% 코드 생성 → 20%만 수동 수정

#### 사례 3: Frontend 컴포넌트 (v0.dev + Cursor)
**기존 방식 (8일)**:
- 디자인 시스템 구축 (2일)
- 컴포넌트 개발 (4일)
- 반응형 대응 (2일)

**AI 방식 (2일)**:
```

## 6. AI 활용 시 주의사항

### 6.1 AI의 한계

| 한계 | 대응 방안 |
|------|----------|
| **비즈니스 로직 오해** | 명확한 요구사항 문서화 필수 |
| **보안 취약점** | 생성된 코드 보안 검토 필수 |
| **트랜잭션 복잡도** | @Transactional 동작 수동 검증 |
| **UX 디테일** | AI 생성 UI 디자이너 검토 |
| **테스트 커버리지** | AI 생성 테스트 + 수동 시나리오 추가 |

### 6.2 사람이 반드시 해야 할 것

1. **비즈니스 로직 검증**: AI가 생성한 입회비 계산 로직이 실제 요구사항과 맞는지 확인
2. **UX 개선**: AI가 만든 UI를 사용자 관점에서 개선
3. **보안 검토**: 인증/인가, SQL Injection 방지 등
4. **성능 최적화**: N+1 쿼리, 인덱스 최적화 등
5. **최종 통합 테스트**: 전체 플로우 시나리오 테스트
